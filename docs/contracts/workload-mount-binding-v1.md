# Contract: Workload Mount Binding V1

Status: P0 review draft

AFSCP creates product-agnostic workload mount bindings. A dedicated external orchestrator consumes privileged mount plans and performs runtime mounting.

## Caller-Visible Binding

Returned to trusted product control planes:

- `mount_binding_id`
- `namespace_id`
- `repo_id`
- `volume_id`
- `mount_path`
- `read_only`
- `status`
- `lease_expires_at`

The caller-visible binding must not include:

- JuiceFS root credentials
- K8s Secret names
- raw absolute filesystem paths
- object store bucket or metadata URLs

## OrchestratorMountPlan

Returned only to caller services with `orchestrator_mount` role:

- `mount_binding_id`
- `volume_id`
- `payload_volume_subdir`
- `mount_path`
- `read_only`
- `secret_ref`
- `security_policy`

`payload_volume_subdir` is relative to the JuiceFS filesystem root, has no leading slash, includes the `afscp/` managed subroot prefix, and is generated by AFSCP from canonical IDs. It points to the repo payload root, never to the repo container directory or JVS control root. `secret_ref` is scoped to the platform mounting component and must not be passed into workload containers.

## Lifecycle

Statuses:

- `issued`
- `pending`
- `active`
- `releasing`
- `released`
- `revoked`
- `expired`
- `failed`

`revoked` is terminal only after the orchestrator confirms that the runtime mount is stopped or otherwise unable to write. A control-plane revoke request that has not yet stopped the runtime remains `releasing` and continues to block restore-run.

P0 requires:

- status update endpoint for the orchestrator
- heartbeat endpoint for live bindings
- release endpoint for normal mount teardown
- revoke endpoint for AFSCP/operator initiated teardown
- `lease_expires_at` on every binding

Read-write bindings in `issued`, `pending`, `active`, or `releasing` state with a live lease count as active writer sessions. Expired read-write bindings block restore-run until reconciliation marks a terminal state. `revoked` may unblock restore-run only when it means confirmed-unmounted, not merely requested.

## Rules

- Product callers create mount bindings after product authorization.
- External orchestrators do not make product authorization decisions.
- Workload containers receive no JuiceFS root credentials.
- K8s RBAC or equivalent runtime policy must prevent ordinary caller ServiceAccounts from reading or referencing JuiceFS root Secrets.
- All AFSCP-managed new repo workload mounts are P0-gated on JVS external control root mode and payload-only mount plans, including read-only mounts.
- Permission-only protection on an embedded `.jvs` directory is insufficient by itself.
- If a repo has embedded JVS control metadata under the mounted root, workload mounts must be rejected until the repo is migrated to external control root mode or protected by a verified filtered mount/view.
- Stock JuiceFS CSI subdirectory mounting and Kubernetes `subPath` may be used to select the payload root. They must not mount the repo container directory that contains both `control/` and `payload/`.
- Read-write binding issuance must be serialized with restore-run by a per-repo writer-session fence.
- AFSCP audits mount binding issuance, orchestrator plan issuance, denial, revoke, and expiry.
