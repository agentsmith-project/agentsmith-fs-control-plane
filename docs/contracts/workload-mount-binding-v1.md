# Contract: Workload Mount Binding V1

Status: GA implementation-baseline contract. FINAL GA is governed by
`docs/GA_RELEASE_GATES.md`, `docs/READINESS_EVIDENCE.md`, and
`scripts/verify-ga-release.sh`.

AFSCP creates product-agnostic workload mount bindings. A dedicated external orchestrator consumes privileged mount plans and performs runtime mounting.

## Caller-Visible Binding

Returned to trusted product control planes:

- `mount_binding_id`
- `namespace_id`
- `repo_id`
- `volume_id`
- `mount_path`
- `read_only`
- `status`
- `lease_expires_at`

The caller-visible binding must not include:

- JuiceFS root credentials
- K8s Secret names
- raw absolute filesystem paths
- object store bucket or metadata URLs

`mount_path` is the destination path inside the workload/container runtime. It
is not a host path, node path, source path, JuiceFS subdir selector, or storage
authorization input. AFSCP must validate it as an absolute container path,
reject empty paths, traversal, control characters, and runtime-reserved paths,
and pass it only as a requested destination to the orchestrator. The
orchestrator remains responsible for runtime-specific path policy before
attaching a mount.

## OrchestratorMountPlan

Returned only to caller services with `orchestrator_mount` role:

- `mount_binding_id`
- `volume_id`
- `payload_volume_subdir`
- `mount_path`
- `read_only`
- `secret_ref`
- `security_policy`

`payload_volume_subdir` is relative to the JuiceFS filesystem root, has no leading slash, includes the `afscp/` managed subroot prefix, and is generated by AFSCP from canonical IDs. It points to the repo payload root, never to the repo container directory or JVS control root. `secret_ref` comes from operator/orchestrator-only runtime configuration keyed by volume ID; AFSCP must not derive it from caller-visible fields or volume ID naming rules. `secret_ref` is scoped to the platform mounting component and must not be passed into workload containers.

`mount_path` in the orchestrator plan has the same container-destination
meaning as the caller-visible binding. It must not be interpreted as a source
path.

## Lifecycle

Statuses:

- `issued`
- `pending`
- `active`
- `releasing`
- `released`
- `revoked`
- `expired`
- `failed`

`released` and `revoked` are evidence-bearing non-accessing terminal statuses: the orchestrator may report them only after confirming that the runtime mount is unmounted or otherwise non-accessing. This also proves unable-to-write. A control-plane revoke request that has not yet stopped runtime access remains `releasing` and continues to block restore-run. `expired` and `failed` are observed terminal statuses; they do not carry confirmed-unmounted, non-accessing, or unable-to-write evidence by themselves.

GA requires:

- status update endpoint for the orchestrator
- heartbeat endpoint for live bindings
- release endpoint for normal mount teardown
- revoke endpoint for AFSCP/operator initiated teardown
- `lease_expires_at` on every binding

Read-write bindings in `issued`, `pending`, `active`, or `releasing` state with a live lease count as active writer sessions. Expired read-write bindings block restore-run until reconciliation reaches an evidence-bearing terminal state proving that the runtime is unmounted or otherwise non-accessing. `revoked` may unblock restore-run only when it means confirmed non-accessing, not merely requested.

Repo lifecycle drain is stricter than restore-run. Any binding, read-only or
read-write, blocks archive, delete, and purge until AFSCP has a confirmed
terminal non-accessing state from the orchestrator or reconciliation process.

Future support for unable-to-write-but-still-mounted/readable evidence for
restore-run must use a new explicit evidence field or status. It must not reuse
`released` or `revoked`, because current GA lifecycle and purge gates treat
those statuses as confirmed non-accessing evidence.

## Rules

- Product callers create mount bindings after product authorization.
- External orchestrators do not make product authorization decisions.
- Workload containers receive no JuiceFS root credentials.
- K8s RBAC or equivalent runtime policy must prevent ordinary caller ServiceAccounts from reading or referencing JuiceFS root Secrets.
- All AFSCP-managed new repo workload mounts are gated on JVS external control root mode and payload-only mount plans, including read-only mounts.
- Permission-only protection on an embedded `.jvs` directory is insufficient by itself.
- If a repo has embedded JVS control metadata under the mounted root, workload mounts must be rejected until the repo is migrated to external control root mode or protected by a verified filtered mount/view.
- Stock JuiceFS CSI subdirectory mounting and Kubernetes `subPath` may be used to select the payload root. They must not mount the repo container directory that contains both `control/` and `payload/`.
- Read-write binding issuance must be serialized with restore-run by a per-repo writer-session fence.
- All binding issuance and status-changing calls must respect the per-repo lifecycle fence.
- Issuance-track mount plans for `issued`, `pending`, or `active` bindings require a fresh lease at the API and store boundary. Expired issuance leases must fail closed until recovery or reconciliation advances state.
- Bindings already in `releasing` may receive the payload-only Secret-bearing orchestrator plan solely so the dedicated orchestrator can identify and tear down existing runtime mounts. This does not re-admit workload access, and AFSCP must force `allow_privileged_workload=false` for these teardown plans.
- AFSCP audits mount binding issuance, orchestrator plan issuance, denial, revoke, and expiry.

## Orchestrator Acceptance Contract

Workload mounts cannot be GA-enabled until the orchestrator contract proves:

- only the dedicated orchestrator identity can fetch mount plans
- ordinary product callers receive no Secret refs, metadata URLs, or raw root paths
- `payload_volume_subdir` is mapped to a verified JuiceFS CSI subdir or controlled Kubernetes `subPath` behavior
- the repo container directory and control root are never mounted
- Secret RBAC prevents ordinary caller ServiceAccounts and workloads from reading JuiceFS root Secrets
- heartbeat extends leases only for non-terminal bindings
- release is idempotent and terminal only after runtime access has ended
- revoke means requested teardown until the orchestrator confirms unmounted or otherwise non-accessing
- `released` and `revoked` are reported only as evidence-bearing non-accessing terminal statuses; `expired` and `failed` remain observed/uncertain and do not unblock lifecycle without future explicit evidence

If these requirements are not met for a runtime, AFSCP returns
`CAPABILITY_DENIED` instead of issuing a degraded mount binding.
