# Security And Tenancy

## Credential Boundary

AFSCP is the only service in the ordinary product path that should hold JuiceFS root metadata and object store credentials.

Forbidden for ordinary flows:

- Returning `metadata_url` to Desktop or Web clients.
- Returning bucket URL, access key, or secret key to ordinary users.
- Injecting JuiceFS credentials into sandbox workload containers.
- Letting sandbox workloads call cloud object store APIs directly.
- Letting user-provided paths select arbitrary filesystem locations.

Allowed:

- K8s Secrets for CSI/Mount Pods managed by platform components.
- Short-lived WebDAV credentials generated by AFSCP and returned through AgentSmith API after authorization.
- Admin/debug direct mount flows behind explicit feature flags.

## Workspace Tenancy

AgentSmith workspace is the storage policy boundary.

Each workspace storage profile may choose:

- AFSCP endpoint.
- Default filesystem.
- Default storage pool.
- Path prefix.
- Quota defaults.
- Export policy.
- Template policy.

Changing a profile should affect new repos only. Existing repos require explicit migration.

## Template Boundary

Template sharing and clone are workspace-scoped only.

Required checks:

- AgentSmith API checks user permission and `template.tenant_workspace_id == request.workspace_id`.
- AFSCP checks source and target repo paths resolve under the same workspace path prefix.
- Even if one user belongs to two workspaces, cross-workspace template clone must be rejected.
- Even if two workspaces use the same storage pool, cross-workspace template clone must be rejected.

## Path Safety

AFSCP must use a single path resolver for WebDAV, sandbox mount, Web file APIs, and JVS operations.

The resolver must reject:

- Absolute paths from callers.
- `..` traversal.
- Symlink escape.
- Empty or malformed repo IDs.
- Workspace/repo mismatches.
- Caller-provided filesystem paths.

## `.jvs` Protection

JVS metadata must be protected from users, agents, and exports.

MVP controls:

- WebDAV must hide or block `.jvs`.
- Sandbox workloads should run as non-root.
- `.jvs` should be owned by an AFSCP-controlled UID with restrictive permissions.
- AFSCP should run `jvs doctor --strict` after sensitive operations.

If sandbox workloads must run as root and can write repo root, filtered mount/view becomes a P0 blocker before enabling JVS-backed writes.

## Operation Safety

AFSCP must persist operation state before executing mutating actions.

Mutating operations include:

- repo create
- repo archive/delete
- save point create
- restore
- repo clone
- export create/revoke
- lifecycle move/rename/detach

Use per-repo operation locks for JVS operations. Do not lock ordinary reads and writes.
