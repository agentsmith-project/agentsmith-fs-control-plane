# Security And Tenancy

## Credential Boundary

AFSCP is the only service in the ordinary storage-control path that should hold JuiceFS root metadata and object store credentials.

Forbidden for ordinary flows:

- Returning `metadata_url` to clients.
- Returning bucket URL, access key, or secret key to ordinary users.
- Injecting JuiceFS credentials into workload containers.
- Letting workloads call cloud object store APIs directly.
- Letting caller-provided paths select arbitrary filesystem locations.

Allowed:

- K8s Secrets or equivalent secret references for CSI/Mount Pods managed by platform components.
- Short-lived WebDAV credentials generated by AFSCP and returned through a calling product after product authorization.
- Admin/debug direct mount flows behind explicit feature flags.

## Namespace Tenancy

AFSCP namespace is the storage isolation boundary.

Each namespace volume binding may choose:

- volume
- isolation class
- logical path namespace policy
- quota defaults
- export policy
- template policy

AFSCP computes the canonical raw filesystem root. Callers do not provide it.

Changing a namespace binding should affect new repos only. Existing repos require explicit migration.

## Template Boundary

Repo template sharing and clone are namespace-scoped in P0.

Required checks:

- Calling product checks product permission before calling AFSCP.
- AFSCP checks source template and target repo resolve under the same namespace.
- Even if the same actor has access to two namespaces, cross-namespace template clone is rejected by default.
- Even if two namespaces use the same volume, cross-namespace template clone is rejected by default.

## Path Safety

AFSCP must use a single path resolver for WebDAV, workload mounts, file APIs, and JVS operations.

The resolver must reject:

- absolute paths from callers
- `..` traversal
- symlink escape
- empty or malformed repo IDs
- namespace/repo mismatches
- caller-provided filesystem paths

## `.jvs` Protection

JVS metadata must be protected from clients, workloads, and exports.

MVP controls:

- WebDAV must hide or block `.jvs`.
- Workloads should run as non-root.
- `.jvs` should be owned by an AFSCP-controlled UID with restrictive permissions.
- AFSCP should run `jvs doctor --strict` after sensitive operations.

If workloads must run as root and can write repo root, filtered mount/view becomes a P0 blocker before enabling JVS-backed writes.

## Operation Safety

AFSCP must persist operation state before executing mutating actions.

Mutating operations include:

- repo create
- repo archive/delete
- save point create
- restore
- repo clone
- template clone
- export create/revoke
- lifecycle move/rename/detach

Use per-repo operation locks for JVS operations. Do not lock ordinary reads and writes.
