# Security And Tenancy

## Credential Boundary

AFSCP is the only service in the ordinary storage-control path that should hold JuiceFS root metadata and object store credentials.

Forbidden for ordinary flows:

- Returning `metadata_url` to clients.
- Returning bucket URL, access key, or secret key to ordinary users.
- Injecting JuiceFS credentials into workload containers.
- Letting workloads call cloud object store APIs directly.
- Letting caller-provided paths select arbitrary filesystem locations.

Allowed:

- K8s Secrets or equivalent secret references for CSI/Mount Pods managed by platform components.
- Short-lived WebDAV credentials generated by AFSCP and returned through a calling product after product authorization.
- Admin/debug direct mount flows behind explicit break-glass controls.

## Caller Service Authorization

Trusted callers are not automatically trusted for every namespace.

For each namespace-bound request, AFSCP must:

- authenticate `caller_service`
- check that the service is allowed on the namespace binding or deployment-level admin/migration policy
- check that the caller role permits the requested operation
- validate namespace/resource consistency
- audit denied authorization, denied namespace, and denied path checks

This prevents confused-deputy failures where a valid internal service token is used against the wrong namespace.

## Namespace Tenancy

AFSCP namespace is the storage isolation boundary.

Each namespace volume binding may choose:

- volume
- isolation class
- allowed caller services and roles
- logical path namespace policy
- quota defaults
- export policy
- mount policy
- template policy

AFSCP computes the canonical raw filesystem root. Callers do not provide it.

Changing a namespace binding should affect new repos only. Existing repos require explicit migration.

## Template Boundary

Repo template sharing and clone are namespace-scoped in P0.

Required checks:

- Calling product checks product permission before calling AFSCP.
- AFSCP checks source template and target repo resolve under the same namespace.
- Even if the same actor has access to two namespaces, cross-namespace template clone is rejected by default.
- Even if two namespaces use the same volume, cross-namespace template clone is rejected by default.

## Path Safety

AFSCP must use a single path resolver for WebDAV, workload mounts, file APIs, and JVS operations.

The resolver must reject:

- absolute paths from callers
- `..` traversal
- symlink escape
- empty or malformed repo IDs
- namespace/repo mismatches
- caller-provided filesystem paths

## `.jvs` Protection

JVS metadata must be protected from clients, workloads, and exports.

P0 access gate:

- WebDAV must hide or block `.jvs`.
- Workload mounts must hide or block `.jvs`, including read-only mounts.
- Permission-only controls are not sufficient by themselves for workload mounts.
- The selected runtime must block lookup, read, write, create, rename, unlink, chmod, chown, hardlink, and symlink operations targeting root-level `.jvs`.
- AFSCP should run `jvs doctor --strict` after sensitive operations.

If these controls cannot be verified for a volume/runtime, workload mounts must be rejected until a filtered mount/view or equivalent gate is implemented.

## Admin/Debug Direct Mount

Direct JuiceFS mount is not an ordinary user or workload access path.

If enabled for operations, it must be:

- disabled by default
- limited to an explicit admin/operator role
- authenticated with mTLS/RBAC or equivalent
- tied to a ticket, reason, namespace, repo, and short TTL
- audited on issue, use, and revoke
- prevented from returning root credentials to ordinary clients

## Operation Safety

AFSCP must persist operation state before executing mutating actions.

Mutating operations include:

- repo create
- save point create
- restore
- repo clone
- template clone
- export create/revoke
- workload mount binding/plan issuance
- lifecycle archive/delete/move/rename/detach in P1

Use operation locks for JVS and lifecycle operations. Do not lock ordinary reads and writes. Restore-run is a control-plane mutation and must acquire a per-repo writer-session fence, reject active read-write export/workload sessions, and block new read-write session issuance in P0 unless an audited operator break-glass flow is explicitly added.
