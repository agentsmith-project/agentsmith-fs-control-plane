# PRD：AgentSmith Workspace 持久文件系统与版本管理能力

## 1. 文档目的

本文档定义 AgentSmith 在沙箱环境中使用持久化工作目录、文件系统版本管理、用户本地访问和服务端统一授权管理的产品需求。

该文档只描述产品方向、功能边界、用户价值和核心流程，不涉及具体驱动、挂载参数、底层协议或部署细节。

---

## 2. 背景

AgentSmith 的 agent 运行在可回收、可重建的沙箱环境中。沙箱本身不应被视为长期数据载体，因为沙箱 Pod 可能因为调度、资源回收、升级、故障等原因随时消失。

但 agent 的工作成果、上下文文件、用户项目文件、环境配置和任务中间状态需要长期保留。因此，AgentSmith 需要一套独立于沙箱生命周期的持久化文件系统能力。

同时，用户希望能够在本地电脑上访问和编辑同一个工作目录；agent 也需要在沙箱中以该目录作为自己的 home 或 workspace 进行工作。除此之外，系统还需要支持类似 save point、快照、恢复、历史查看等版本管理能力。

因此，AgentSmith 需要建设一个统一的 **Workspace Storage 产品能力**，让文件系统成为 AgentSmith 的核心状态载体之一。

## 3. 产品目标

### 3.1 核心目标

AgentSmith 应支持为每个 repo / workspace 提供一个持久、可恢复、可授权访问的文件系统空间。

该空间应同时满足：

1. **沙箱可挂载**
   AgentSmith 的 agent 可以在沙箱中将指定 workspace 作为 home 或工作目录使用。

2. **用户本地可访问**
   用户可以在个人电脑上访问指定 workspace，像访问远程文件夹一样查看、编辑和管理文件。

3. **数据持久化**
   沙箱销毁、重建或迁移后，workspace 中的文件状态仍然保留。

4. **版本可管理**
   用户或系统可以对 workspace 创建 save point，查看历史状态，并在需要时恢复到指定状态。

5. **凭据不暴露**
   用户和沙箱都不应直接获得底层存储系统的敏感凭据。

6. **权限可控**
   用户、agent、沙箱和外部访问入口都必须经过 AgentSmith 的统一授权。

7. **操作可审计**
   文件系统挂载、导出、保存、恢复、删除等关键动作都应被记录。

---

## 4. 非目标

本阶段不追求以下能力：

1. 不提供多人实时协作编辑能力。
   同一 workspace 可以被多个端访问，但默认不承诺类似在线协作文档的实时冲突解决能力。

2. 不让终端用户直接管理底层存储。
   用户只访问被授权的 workspace，不直接操作底层文件系统、对象存储、元数据服务或系统级凭据。

3. 不把沙箱作为版本管理权威。
   沙箱可以读写文件，也可以请求创建 save point，但最终的版本管理操作应由 AgentSmith 服务端统一执行。

4. 不默认支持任意多方同时写入同一 workspace。
   并发写入可以作为后续协作模式，但不作为第一版默认能力。

5. 不在第一版解决所有复杂冲突合并问题。
   如果用户 PC 和 agent 同时修改同一文件，系统不负责自动语义合并。

## 5. 核心概念

### 5.1 Repo

Repo 是一个文件系统版本管理单元。

一个 repo 下可以有多个 workspace。Repo 负责承载历史、save point、恢复记录和版本管理上下文。

示例：

```text
repo: project-alpha
```

### 5.2 Workspace

Workspace 是用户和 agent 实际操作的文件目录。

一个 repo 可以有多个 workspace，例如：

```text
main
experiment-a
debug-session-1
```

其中 `main` 是默认 workspace。

### 5.3 Home Workspace

当 workspace 被挂载到沙箱中作为 agent 的 home 目录时，该 workspace 即成为 agent 的持久 home。

这意味着 agent 在 home 目录内创建、修改、删除的文件，都应保存在 workspace 中，而不是保存在临时沙箱中。

### 5.4 Save Point

Save point 是 workspace 在某一时刻的保存状态。

用户或系统可以通过 save point：

1. 标记一个稳定状态。
2. 查看历史版本。
3. 从历史状态恢复。
4. 基于历史状态创建新的 workspace。

### 5.5 Export

Export 表示将一个 workspace 暴露给用户本地电脑访问。

Export 应由 AgentSmith 创建、授权、管理和撤销。用户通过 export 访问 workspace，但不直接获得底层存储凭据。

### 5.6 Attachment

Attachment 表示将一个 workspace 挂载给某个沙箱使用。

Attachment 应由 AgentSmith 控制，沙箱只获得最终文件视图，不获得底层系统管理权限。

## 6. 用户角色

### 6.1 普通用户

普通用户希望：

1. 创建或选择一个 repo。
2. 在默认 workspace 中工作。
3. 让 agent 在该 workspace 中执行任务。
4. 在本地电脑访问同一个 workspace。
5. 查看历史 save point。
6. 在需要时恢复到历史状态。

### 6.2 Agent

Agent 希望：

1. 在稳定的 home 目录中工作。
2. 即使沙箱被回收，也能继续访问之前的文件状态。
3. 将任务产物保存到 workspace。
4. 在用户授权下读写项目文件。

### 6.3 沙箱管理器

沙箱管理器负责：

1. 接收任务请求。
2. 确定使用哪个 repo 和 workspace。
3. 创建沙箱。
4. 将 workspace 作为 home 或工作目录提供给沙箱。
5. 在沙箱结束后释放运行资源，但不删除 workspace 数据。

### 6.4 Workspace 管理服务

Workspace 管理服务负责：

1. 管理 repo 和 workspace。
2. 管理 save point。
3. 管理 restore 操作。
4. 管理用户本地访问。
5. 管理沙箱挂载。
6. 管理权限、租约、审计和生命周期。

### 6.5 系统管理员

系统管理员希望：

1. 查看各 workspace 的使用情况。
2. 设置容量、保留策略和访问策略。
3. 审计用户和 agent 的关键操作。
4. 在异常情况下恢复或隔离 workspace。
5. 管理底层存储，但不暴露给普通用户。

---

## 7. 核心用户场景

### 7.1 用户创建项目并让 agent 工作

用户创建一个 repo，系统自动生成默认 workspace。

用户启动 agent 后，AgentSmith 将该 workspace 作为沙箱中的 home 或工作目录。Agent 在其中读取项目文件、生成文件、保存中间状态和任务结果。

即使沙箱被回收，用户稍后重新启动 agent 时，agent 仍能看到之前的文件状态。

---

### 7.2 用户在本地电脑访问 agent 的工作目录

用户希望在本地电脑上查看 agent 修改过的文件。

用户在 AgentSmith 中选择某个 workspace，并创建一个本地访问入口。系统只暴露该 workspace 的文件视图，并根据用户权限控制读写能力。

用户可以在本地打开、编辑或下载文件，但不会看到底层存储凭据，也不能访问其他 workspace。

---

### 7.3 用户创建 save point

当 agent 完成阶段性任务后，用户希望保存当前状态。

用户点击“创建 save point”，系统记录当前 workspace 的状态，并生成一条历史记录。

之后用户可以查看该 save point 的说明、创建时间、创建者和关联任务。

---

### 7.4 用户恢复到历史状态

用户发现当前 workspace 出现问题，希望恢复到之前的 save point。

系统先展示恢复预览，说明恢复会影响哪些文件。用户确认后，系统执行恢复。

恢复操作期间，系统应避免其他写入者继续修改 workspace，防止恢复结果不一致。

---

### 7.5 用户基于历史状态创建新 workspace

用户希望从某个历史状态继续探索另一条路径。

用户选择一个 save point，并创建新的 workspace。新 workspace 拥有独立的当前文件状态，用户可以让 agent 在其中继续工作，而不影响原来的 main workspace。

---

### 7.6 沙箱被回收后恢复工作

agent 正在工作时，沙箱因为资源调度被回收。

AgentSmith 重新创建沙箱，并将同一个 workspace 再次提供给沙箱。agent 可以继续使用之前的 home 文件状态。

用户不需要关心沙箱在哪个节点运行，也不需要重新同步文件。

## 8. 产品功能需求

## 8.1 Repo 管理

### 需求

系统应支持创建、查看、删除和管理 repo。

每个 repo 应有唯一标识、所属用户或组织、默认 workspace、创建时间、状态和权限信息。

### 目标

Repo 是用户项目文件和版本历史的长期容器。

### 验收标准

1. 用户可以创建 repo。
2. 每个 repo 默认拥有一个 main workspace。
3. 用户只能查看自己有权限访问的 repo。
4. 删除 repo 必须有明确确认流程。
5. 删除 repo 不应因为误操作立即导致不可恢复的数据丢失。

---

## 8.2 Workspace 管理

### 需求

系统应支持在 repo 下创建和管理多个 workspace。

Workspace 可以来自空目录、默认模板、已有 workspace 或历史 save point。

### 目标

Workspace 是 agent 和用户实际工作的文件空间。

### 验收标准

1. 用户可以查看 repo 下的 workspace 列表。
2. 用户可以选择某个 workspace 作为 agent 的工作目录。
3. 用户可以基于 save point 创建新 workspace。
4. Workspace 应支持状态标记，例如正常、挂载中、恢复中、只读、归档中等。
5. Workspace 的生命周期不依赖任何单个沙箱。

---

## 8.3 沙箱挂载

### 需求

系统应支持将指定 workspace 提供给沙箱作为 home 或工作目录。

沙箱启动时，沙箱管理器应明确指定 repo、workspace 和挂载目的。

### 目标

让 agent 在一个持久化、可恢复、可版本管理的文件空间中工作。

### 验收标准

1. 沙箱可以使用指定 workspace 作为 home。
2. 沙箱销毁后，workspace 数据仍然存在。
3. 沙箱重建后，可以重新使用同一个 workspace。
4. 用户和系统可以看到 workspace 当前是否被某个沙箱使用。
5. 系统可以阻止不安全的重复写入。

---

## 8.4 用户本地访问

### 需求

系统应支持将指定 workspace 暴露给用户本地电脑访问。

用户访问时只应看到被授权 workspace 的文件内容，不应看到底层存储凭据、系统控制目录或其他 workspace。

### 目标

让用户可以像访问远程文件夹一样访问 agent 的工作目录。

### 验收标准

1. 用户可以为指定 workspace 创建本地访问入口。
2. 本地访问入口可以设置只读或读写。
3. 本地访问入口可以被撤销。
4. 本地访问入口可以设置有效期。
5. 用户不能通过本地访问入口越权访问其他 workspace。
6. 用户不能看到底层存储系统的敏感信息。

---

## 8.5 Save Point 管理

### 需求

系统应支持为 workspace 创建 save point，并记录 save point 的基本信息。

### 目标

让用户和系统可以保存阶段性成果，便于回溯、恢复和分支探索。

### 验收标准

1. 用户可以手动创建 save point。
2. 系统可以在关键任务节点自动创建 save point。
3. 每个 save point 应包含创建者、创建时间、说明、关联 workspace 和状态。
4. 用户可以查看 save point 历史。
5. 用户只能查看自己有权限访问的 save point。

---

## 8.6 Restore 管理

### 需求

系统应支持将 workspace 恢复到指定 save point。

恢复前应提供预览，恢复中应阻止不安全的并发写入。

### 目标

让用户可以安全地回到历史状态。

### 验收标准

1. 用户可以选择 save point 并发起恢复预览。
2. 系统应展示恢复会影响的文件范围。
3. 用户确认后，系统才执行恢复。
4. 恢复期间，系统应阻止新的写入访问。
5. 恢复失败时，系统应保留操作记录并支持后续处理。
6. 恢复完成后，workspace 状态应清晰可见。

---

## 8.7 Workspace 访问模式

### 需求

系统应支持不同的 workspace 访问模式。

建议第一版支持：

1. **独占读写**
   同一时间只允许一个主要写入者，例如一个沙箱或一个用户本地读写入口。

2. **共享只读**
   多个用户或沙箱可以只读查看 workspace。

3. **受控共享读写**
   可作为后续增强能力，用于允许用户和 agent 同时写入，但需要明确风险和冲突处理策略。

### 目标

在灵活访问和数据一致性之间取得平衡。

### 验收标准

1. 系统可以知道当前 workspace 的活跃访问者。
2. 系统可以区分只读和读写访问。
3. 系统可以拒绝不安全的并发写入。
4. 恢复操作期间，系统可以阻止新的读写访问。
5. 管理员可以查看 workspace 的访问状态。

---

## 8.8 权限与授权

### 需求

所有 repo、workspace、save point、export 和 sandbox attachment 都必须经过授权。

### 目标

确保用户只能访问自己被授权的数据，agent 只能访问任务允许的数据。

### 验收标准

1. 用户只能看到自己有权限的 repo 和 workspace。
2. 用户只能为有权限的 workspace 创建本地访问入口。
3. 沙箱只能挂载任务指定的 workspace。
4. 用户和沙箱不能获取底层存储凭据。
5. 关键操作需要记录操作人和操作来源。
6. 管理员可以撤销用户或沙箱的访问权限。

---

## 8.9 审计与操作记录

### 需求

系统应记录关键操作。

包括但不限于：

1. 创建 repo。
2. 创建 workspace。
3. 启动沙箱并挂载 workspace。
4. 创建本地访问入口。
5. 撤销访问入口。
6. 创建 save point。
7. 执行 restore。
8. 删除或归档 workspace。
9. 权限变更。
10. 异常恢复。

### 目标

为安全、合规、排障和用户信任提供基础。

### 验收标准

1. 每个关键操作都有审计记录。
2. 审计记录包含操作人、时间、对象、结果和来源。
3. 管理员可以查询审计记录。
4. 用户可以查看与自己 workspace 相关的关键历史。
5. 审计记录不能被普通用户修改。

---

## 8.10 生命周期管理

### 需求

系统应管理 repo、workspace、save point、export 和 sandbox attachment 的生命周期。

### 目标

避免数据误删、资源泄露和长期无主资源。

### 验收标准

1. 沙箱结束不会自动删除 workspace。
2. 本地访问入口过期后应自动失效。
3. Workspace 删除应有保护机制。
4. Save point 可根据策略保留或清理。
5. 管理员可以查看长期未使用的 workspace。
6. 系统支持归档而不是直接删除。

---

## 8.11 容量与配额

### 需求

系统应支持对用户、repo 或 workspace 设置容量限制和使用量统计。

### 目标

防止单个用户或 workspace 消耗过多存储资源。

### 验收标准

1. 用户可以看到 workspace 的大致使用量。
2. 管理员可以设置用户或 workspace 的容量限制。
3. 超出限制时，系统应限制新的写入或给出明确提示。
4. 系统应支持使用量告警。
5. 系统应支持清理或归档策略。

---

## 9. 关键产品原则

### 9.1 文件系统是 AgentSmith 的长期状态

沙箱是计算资源，workspace 是状态资源。

沙箱可以被销毁，workspace 不能因为沙箱销毁而丢失。

---

### 9.2 用户看到 workspace，不看到底层存储

用户本地电脑访问的是被授权的 workspace 文件视图。

用户不应该看到：

1. 底层存储凭据。
2. 系统控制目录。
3. 其他 repo 或 workspace。
4. 版本管理内部数据。
5. AgentSmith 服务端管理信息。

---

### 9.3 Agent 可以工作，但不能成为管理者

Agent 可以读写 workspace 文件，但不应直接管理版本历史、恢复操作或系统级访问权限。

版本管理、恢复和导出应由 AgentSmith 服务端统一执行。

---

### 9.4 默认单写者，多读者

为了保证 save point 和 restore 语义清晰，第一版应默认：

```text
一个 workspace 同一时间最多一个主要写入者
多个只读访问者可以并存
```

后续可以支持共享读写，但应明确冲突风险。

---

### 9.5 恢复操作必须受控

Restore 是高风险操作，必须由服务端执行，并且需要：

1. 权限检查。
2. 恢复预览。
3. 用户确认。
4. 写入锁定。
5. 操作审计。
6. 失败恢复机制。

---

## 10. MVP 范围

### 10.1 MVP 必须支持

1. 创建 repo。
2. 自动创建默认 workspace。
3. 将 workspace 挂载给沙箱作为 home。
4. 沙箱销毁后 workspace 数据保留。
5. 用户可以为 workspace 创建本地访问入口。
6. 用户可以创建 save point。
7. 用户可以查看 save point 历史。
8. 用户可以从 save point 恢复 workspace。
9. 系统支持基本权限控制。
10. 系统支持基本审计。
11. 系统支持默认独占读写策略。
12. 系统不暴露底层存储凭据给用户或沙箱。

---

### 10.2 MVP 可以暂缓

1. 多人实时协作。
2. 自动冲突合并。
3. 跨区域同步。
4. 高级配额计费。
5. 复杂组织权限模型。
6. 大规模 gateway 池化。
7. Workspace 自动分支推荐。
8. 高级数据治理能力。
9. 细粒度文件级权限。
10. 用户自定义版本保留策略。

## 11. 推荐产品流程

## 11.1 启动 agent 流程

```text
用户选择 repo 和 workspace
        ↓
AgentSmith 检查权限
        ↓
系统确认 workspace 当前可写
        ↓
创建沙箱
        ↓
将 workspace 提供给沙箱作为 home
        ↓
agent 在持久 workspace 中工作
        ↓
沙箱结束，workspace 保留
```

---

## 11.2 用户本地访问流程

```text
用户选择 workspace
        ↓
用户选择访问模式：只读或读写
        ↓
AgentSmith 检查权限和当前访问状态
        ↓
系统创建本地访问入口
        ↓
用户在本地电脑访问 workspace
        ↓
访问过期或用户主动撤销
```

---

## 11.3 创建 save point 流程

```text
用户或系统发起 save point 请求
        ↓
AgentSmith 检查权限
        ↓
系统确认 workspace 状态安全
        ↓
创建 save point
        ↓
记录历史和审计
        ↓
返回 save point 信息
```

---

## 11.4 恢复流程

```text
用户选择 save point
        ↓
系统生成恢复预览
        ↓
用户确认
        ↓
系统锁定 workspace 写入
        ↓
执行恢复
        ↓
记录审计
        ↓
恢复完成后释放 workspace
```

## 13. 风险与产品决策

### 13.1 并发写入风险

如果用户本地电脑和 agent 同时写入同一个 workspace，可能出现文件覆盖、状态不一致或 save point 不准确。

### 产品决策

第一版默认采用独占写入模式。共享写入作为高级功能，后续再支持。

---

### 13.2 恢复操作风险

Restore 会改变 workspace 当前文件状态。

### 产品决策

恢复必须经过预览和确认。恢复过程中应阻止其他写入者。

---

### 13.3 误删风险

Workspace 是长期数据资源，误删会造成严重后果。

### 产品决策

删除应默认转为归档或软删除。永久删除需要更高权限和二次确认。

---

### 13.4 凭据泄露风险

如果用户或沙箱拿到底层存储凭据，可能绕过 AgentSmith 权限系统。

### 产品决策

用户和沙箱永远只访问 workspace 文件视图，不获得底层存储凭据。

---

### 13.5 版本管理控制区风险

如果用户或 agent 可以直接修改版本管理内部数据，历史状态可能被破坏。

### 产品决策

版本管理内部数据必须由服务端控制，不应暴露给普通用户或沙箱。

## 16. 总结

AgentSmith 需要建设的不是一个简单的“远程目录挂载”能力，而是一个完整的 **Workspace Storage 产品系统**。

这个系统的核心价值是：

```text
让 agent 在可回收沙箱中拥有持久 home
让用户可以在本地访问同一个 workspace
让 workspace 可以被保存、恢复和分支
让所有访问都由 AgentSmith 统一授权
让底层存储和版本管理细节不暴露给用户
```

推荐的产品方向是：

1. Repo 承载版本管理上下文。
2. Workspace 承载用户和 agent 实际工作文件。
3. Sandbox 将 workspace 作为持久 home 使用。
4. 用户 PC 通过受控入口访问 workspace。
5. Save point 和 restore 由服务端统一管理。
6. 默认采用单写者、多读者模型。
7. 所有高风险操作都要有权限控制和审计。

最终目标是让 AgentSmith 的文件系统成为一个可靠、可恢复、可访问、可审计的长期工作空间，而不是依附于短生命周期沙箱的临时目录。
