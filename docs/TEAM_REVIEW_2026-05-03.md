# Team Review Closure: 2026-05-03

Status: historical review closure; superseded where noted by the 2026-05-04 JVS external-control update.

2026-05-04 update: JVS now supports external control roots. Current AFSCP handoff docs use a separated `control/.jvs` plus `payload/` repo layout. Historical `.jvs` exposure findings in this document still apply to embedded-control repos, but not to new AFSCP-managed repos that mount/export only `payload/`.

This review used product, platform architecture, security/operations, and API/handoff perspectives. The goal was to prevent early implementation choices that would force rework.

## Overall Product Conclusion

The decoupled direction is correct.

AFSCP should remain a robust functional storage substrate. It should manage volumes, namespaces, repos, repo templates, exports, workload mount bindings, orchestrator mount plans, JVS execution, operations, logs, and audit. It should not understand notebook tasks, file libraries, product workspaces, projects, template catalog UX, or user-facing product permission models.

The main adjustment from review is that AFSCP still needs storage-control authorization. It should not decide product permissions, but it must know which caller services may operate which namespaces and which roles those services have.

## Findings Closed Into The Handoff

### 1. Caller Service Authorization Is P0

Risk: a trusted internal caller token could be used against the wrong namespace.

Decision:

- namespace binding includes `allowed_callers`
- AFSCP checks `caller_service` and role before each namespace-bound operation
- denied authorization is audited

### 2. Workload Mount Secret Boundary Must Be Split

Risk: returning `secret_ref` and raw subdir values to product callers expands the credential boundary and couples callers to mount internals.

Decision:

- product caller receives `WorkloadMountBinding`
- dedicated orchestrator receives `OrchestratorMountPlan`
- `secret_ref` is orchestrator-only
- `payload_volume_subdir` is relative to the JuiceFS filesystem root, includes the `afscp/` managed subroot, points to the repo payload root, and is generated by AFSCP
- mount bindings are lease-based and report status through orchestrator heartbeat/release/revoke flows

### 3. Workload Mounts Need A JVS Control Metadata Protection Gate

Risk: if JVS metadata sits inside the mounted repo root, root or privileged workloads could tamper with `.jvs`.

Decision:

- New AFSCP-managed repos use JVS external control roots.
- WebDAV and workload mounts expose only the repo payload root.
- permission-only controls are insufficient because the writable parent directory can allow entry-level operations
- if a legacy embedded-control repo cannot be migrated or protected by a verified filtered view, AFSCP rejects workload mounts

### 4. Ordinary Concurrent IO Remains Allowed, But Restore-Run Is Fenced

Risk: restore-run racing active writers can produce an unsafe version state.

Decision:

- AFSCP does not enforce a product-level single-writer model for ordinary file IO
- P0 restore-run acquires a per-repo writer-session fence, blocks new read-write session issuance, then rejects active read-write export/workload sessions by default
- operator break-glass restore is P1 unless explicitly pulled forward with audit and revoke/drain controls

### 5. Repo Lifecycle Is P1

Risk: archive/delete/rename/detach require active session drain and recovery semantics, which would expand MVP.

Decision:

- P0 supports repo create/get, save/history/restore-run, template create/clone, export, and mount binding
- archive/delete/rename/detach remain P1

### 6. Operation Recovery Needs Concrete Fields And Matrix

Risk: crash recovery after clone/export/restore-run cannot be deterministic with only generic states.

Decision:

- operation records include `phase`, `attempt`, lease fields, request hash, external resource IDs, verification result, and compensation status
- P0 operation matrix defines lock and recovery strategy by operation type

### 7. Template Semantics Are Immutable And Namespace-Scoped In P0

Risk: template update/delete/catalog semantics could leak product-specific workflow into AFSCP or create cross-volume surprises.

Decision:

- P0 `RepoTemplate` is immutable after publication
- changing content means creating a new template/revision owned by the caller's catalog
- template creation always creates a fresh source save point in the operation; caller-provided historical save point templating is P1
- template clone history mode must be pinned to JVS capability; `all` requires durable imported-save-point protection, otherwise use `main`
- cross-namespace clone is rejected
- cross-volume clone is rejected with `VOLUME_MISMATCH_REQUIRES_IMPORT` in P0; no silent clone into the old template volume

### 8. Path Resolver And WebDAV Need Protocol-Level Tests

Risk: encoded paths, symlinks, hardlinks, WebDAV MOVE/COPY/PROPFIND, and root-level `.jvs` creation attempts can bypass simple filters.

Decision:

- strict ID grammar
- decode-once validation
- shared resolver test corpus
- WebDAV method-level policy
- read-only WebDAV denies mutating methods

Updated docs:

- [contracts/repo-path-contract-v1.md](contracts/repo-path-contract-v1.md)

### 9. Migration Requires Freeze Or Delta Sync

Risk: legacy migration can lose writes if source remains writable without delta/final lock.

Decision:

- migration is not automatic in P0
- explicit migration requires source read-only/maintenance mode or initial-copy plus delta-sync plus final write freeze
- rollback depends on preserving source generation and cutover metadata

## Remaining Handoff Gates

Before endpoint implementation:

- choose runtime/framework ADR
- create JSON schemas and generate internal OpenAPI
- pin supported JVS version or commit
- freeze exact JVS argv/JSON/error mapping
- confirm orchestrator can consume payload-only mount plans and report mount binding heartbeat/release
- confirm auth provider maps to canonical `caller_service`
- confirm writer-session fence semantics

These are gates, not expanded product scope.
